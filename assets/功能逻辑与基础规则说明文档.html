一、全局架构与设计原则

1.1 角色体系

平台支持三种角色切换，定义在 state.role 中，通过顶部角色开关控制：

角色标识首页路由典型操作
品牌方（广告主）brandpromotions创建活动、管理达人、审核交付
创作者（达人）creatoropportunityHall查看任务、报名、提交产物
运营方（管理员）operatoradminDashboard审核活动/报名/产物、管理用户

1.2 路由机制

使用 URL hash 实现页面切换。核心函数 go(page) 设置 window.location.hash = page 触发 hashchange 事件，驱动 render() 重新渲染。当页面不匹配任何已知页面时，降级渲染 promotions。

1.3 全局状态管理

所有状态存储在全局 state 对象中，无状态管理框架。关键全局字段包括：

字段路径类型说明
state.role"brand" | "creator" | "operator"当前角色
state.pagestring当前页面路由 ID
state.brandNotifications{ unreadCount, items[] }广告主通知数据
state.creatorNotifications{ unreadCount, items[] }达人通知数据
state.tabs.*Record<string, string>各页面 Tab 切换状态
state.*Filtersobject各类筛选项配置
state.appliedCampaignsstring[]达人已报名的任务列表
设计约束：当前为纯前端原型，无后端 API 调用。所有数据存储于硬编码常量或运行时 state 中，页面切换不丢失数据（DOM 重新渲染但 JS 状态持久）。

1.4 交互事件体系

采用全局事件委托 document.addEventListener("click", ...) 处理三类交互：

  • 导航：[data-nav="pageId"] — 点击后调用 go(pageId)
  • Tab 切换：[data-tab] — 更新 state.tabs[tab] = value 后重新渲染
  • 动作：[data-action="actionName"] — 路由到 handleAction(actionName, target)，约 200+ 分支

1.5 弹窗与抽屉体系

组件容器控制函数关闭方式
Modal 弹窗#modalRootopenModal(content, small?, extraClass?)点击蒙层 / closeOverlay()
Drawer 抽屉#modalRoot各 open*Drawer() 函数点击蒙层 / closeOverlay()
Toast 提示#toastRoottoast(message, title?)2.6 秒自动消失

二、广告主 · 活动管理（Promotions）

2.1 页面概述

广告主工作台首页，集中查看所有推广活动的状态、预算与进展，并支持创建新活动。

核心模块：KPI 指标卡 → 筛选栏 → 被驳回活动队列 → 活动列表表格

2.3 数据与接口推导

数据字段来源对应接口字段（推导）说明

campaigns[].budget硬编码campaign_budget预算金额（USD）

2.4 异常与边界场景

  • 空数据：brandHasCampaigns === false 时展示三步引导流程，不显示列表
  • 筛选无结果：展示空状态提示"暂无匹配结果"
  • 删除操作：当前仅弹 Toast 确认，未实现真实删除逻辑

2.5 待确认逻辑

⚠️ 活动状态机：当前状态包括"进行中/草稿/已到期/已驳回/审核中"，但状态流转规则尚未定义（例如：草稿→审核中→进行中/已驳回 的完整流转链路需后端配合）。建议需明确：创建活动后是否立即进入"审核中"；被驳回后重提是否重置状态为"审核中"；活动到期后是否有自动状态变更触发。

三、广告主 · 创建活动（PublishTask）

3.1 页面概述

广告主填写推广信息、配置平台达人要求与预算，创建可报名的合作任务。

核心模块：一页式表单（推广信息 → 平台配置 → 交付信息 → 匹配达人）

3.2 核心交互与基础逻辑

字段/元素类型交互动作业务处理规则

达人标签多选下拉点击展开 + checkbox10 种达人标签可选，支持多选；按平台单独配置
合作类型chips 多选点击切换 active 状态选项：全片/插片/直播/长视频/Shorts/社区post/素材授权

结算方式select选择选项：固定价/固定CPM/按曝光量区间现金结算
结算周期select选择选项：按7天/3天/14天实际曝光量结算
期望效果select选择选项：曝光/点击/转化

确认发布primary-button点击校验通过后调用 submit-publish，将活动加入 campaigns 列表

2.3 平台配置完整性校验

publishPlatformConfigStatus(config) 检查 5 个必填项：

  1. 粉丝范围（followerMinK + followerMaxK）
  2. 达人标签（creatorTypes 非空）
  3. 合作类型（cooperationTypes 非空）
  4. 内容要求（contentRequirement 非空）
  5. 平台预算（platformBudget 非空）

完成 0 项 → "未配置"（灰色）、1-4 项 → "待补充"（黄色）、全部完成 → "已完成"（绿色）。

2.4 数据与接口推导

数据字段推导接口字段说明
publishDraft.campaignNamecampaign_name活动名称
publishDraft.platforms[]platforms[]选中平台列表
publishDraft.countries[]target_countries[]推广国家列表
publishDraft.platformConfigs[platform]platform_configs[]每个平台的细分配置（JSON）
publishDraft.registrationRangeregistration_start/end达人报名时间范围
publishDraft.projectDeadlineproject_deadline项目截止日期

2.5 异常与边界场景

  • 必填字段校验：canOpenPublishMatch() 校验 7 个基础字段和平台配置完整性，任一未填则匹配达人按钮禁用
  • 平台联动：切换平台 Tab 时自动保存当前平台配置到 platformConfigs
  • 重复提交：发布后活动插入 campaigns 数组，未做去重校验

2.6 待确认逻辑

⚠️ 表单验证时机：当前无"提交前全量校验"逻辑，仅在"查看匹配达人"时做部分校验。建议在"确认发布"时增加全量必填校验。此外，草稿保存未实现持久化：刷新页面后草稿数据丢失。

数据字段来源推导接口说明

4.4 异常与边界场景

  • 报名达人为空：creators 数组遍历展示，无需特殊空状态（当前始终有 8 条数据）
  • 已合作达人为空：筛选后无匹配数据时展示空状态
  • 确认/拒绝报名：当前仅为模拟操作，未实际更新数据源

4.5 待确认逻辑

⚠️ 报名审核流程：广告主端的"确认/拒绝"操作应同步更新达人端 applicationRecords 中的 statusKey，并触发达人通知，但当前实现仅为 Toast 占位。建议后端接口统一处理报名审核的原子性操作和双向状态同步。

报价校验规则

场景广告主设置达人输入校验规则
固定价模式单人单价最低 = 最高（即锁定金额）输入框 readonly，展示固定金额无需用户填写，直接取广告主设定值
区间价模式单人单价最低 ≠ 最高（设定范围）输入框可编辑达人填写的报价必须 ≥ 最低价 且 ≤ 最高价；超出范围则弹 Toast 提示
toast("报名报价需填写在 $X - $Y 之间")，报名不提交

对应代码逻辑：taskQuoteSpec(taskName) 根据任务名称从 creatorCampaigns 中查找报价区间；submit-apply 分支中两次校验：非空校验 + 范围校验，均不通过时阻止报名提交。

联动关系：报价区间数据来源于广告主在创建活动时为每个推广平台配置的 unitPriceMin / unitPriceMax 字段（USD）。达人报名时按任务级取配置，不区分平台维度。

20.4 审核操作的数据流转

活动审核 — approve:
  → adminReviewHistoryRows.unshift({ result:"已通过", ... })
  → campaigns[].status → "进行中"
  → adminAdvertiserCampaignRows[].status → "招募中"
  → creatorCampaigns 添加任务（可见于达人机会大厅）
  → brandNotifications 添加通知

活动审核 — reject:
  → adminReviewHistoryRows.unshift({ result:"已拒绝", ... })
  → campaigns[].status → "已驳回"
  → adminAdvertiserCampaignRows[].status → "已拒绝"
  → brandNotifications 添加通知

产物审核 — approve:
  → adminDeliverableReviewDetails[].status → "通过"
  → adminCampaignFulfillmentRows[].deliverableStatus → "待发布"
  → collaborations[].progress → "待发布"
  → creatorNotifications 添加通知
